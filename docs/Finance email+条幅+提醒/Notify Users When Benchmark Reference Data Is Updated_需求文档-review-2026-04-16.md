# 需求文档审核报告

## 文档概况

- **文档类型**：PRD（产品需求文档）——包含功能概述、详细使用流程、触发机制和特殊情况说明，但缺少标准 PRD 的部分章节
- **审核日期**：2026-04-16
- **文档摘要**：当外部行业基准或 LG 内部基准发生变化时，系统通过邮件和应用内横幅向 Portfolio Manager 及 Company Admin 发出通知

## 审核概览

| 严重程度 | 数量 |
|---------|------|
| 🔴 严重  | 4 个 |
| 🟡 警告  | 9 个 |
| 🔵 建议  | 5 个 |
| **合计** | **18 个** |

**总体评价**：文档对主流程描述较为清晰，两条工作流的触发条件和通知方式均有说明。但存在多处关键定义缺失（角色权限范围不明、阈值示例与规则矛盾），以及邮件与横幅在多触发场景下的合并逻辑不完整，需在开发前补充澄清，否则极易导致实现偏差和返工。

## 详细问题清单

### 一、整体结构问题

**[S-01] 🟡 缺少非功能性需求章节**
- **位置**：全文
- **问题**：文档未定义性能要求（如每日扫描耗时上限、邮件发送延迟 SLA）、可靠性要求（邮件送达失败的重试机制）及数据量级预估（监测公司数量、指标数量），对后端架构设计存在遗漏。
- **建议**：补充非功能性需求章节，至少包含：邮件发送时效要求、每日监测任务的最大执行时间窗口、通知失败的容错机制描述。

**[S-02] 🟡 关键术语缺少统一定义**
- **位置**：全文
- **问题**：文档中使用了 "closed month"、"百分位"、"平台基准"、"同行基准"、"internal peer" 等专业术语，但未提供术语表或跳转说明，不同背景的读者（前端、后端、QA）可能产生不同理解。
- **建议**：在文档末尾增加「术语说明」节，或标注各专业术语的跳转引用。

**[S-03] 🔵 文档缺少版本及元数据信息**
- **位置**：文档头部
- **问题**：无版本号、作者、审核人、创建/更新日期，无法追溯文档历史变更，也不便于团队协作中明确文档状态。
- **建议**：在文档顶部增加元数据块（版本号、作者、最后更新日期、状态）。

---

### 二、逐条需求问题

**[R-01] 🔴 "其他有该公司权限的人"范围未定义，影响开发边界**
- **位置**：2.2 节横幅提示、邮件提示
- **维度**：完整性
- **问题**：2.2 工作流中多次出现"Portfolio manager 或其他有该公司权限的人"，但"其他有该公司权限的人"具体指哪些角色（如 Analyst、Viewer 等）完全未定义，导致通知接收人的代码逻辑无法确定，且该范围在 2.1 与 2.2 中亦不一致（2.1 横幅仅提及 Portfolio Manager，未提及其他有权限人员）。
- **建议**：明确系统中所有角色列表，标注每个角色在 2.1 和 2.2 中是否接收横幅通知和邮件通知，并保持两个工作流的接收人范围定义一致或说明差异原因。

**[R-02] 🔴 横幅在后续新一轮触发时是否重新显示未定义**
- **位置**：2.1 节横幅提示第 4 点、2.2 节横幅提示第 3 点
- **维度**：完整性
- **问题**：文档说明"用户可关闭横幅，若不关闭则一直显示"，但未定义：当用户已关闭横幅后，若再次触发新一轮通知（例如，用户关闭了 KeyBanc 2026 横幅后，再次添加 KeyBanc 2027），横幅是否重新弹出？关闭状态是否与特定触发事件绑定还是全局绑定？此逻辑直接影响前端状态管理和后端通知记录的数据模型设计。
- **建议**：补充横幅关闭状态的作用域定义：说明关闭记录是与单次触发事件绑定还是全局一次性关闭，以及后续新触发是否重置关闭状态。

**[R-03] 🔴 2.1 多平台同时更新时邮件合并逻辑未定义**
- **位置**：2.1 节邮件提示
- **维度**：完整性
- **问题**：横幅提示中明确了"若多个平台版本有更新，用逗号间隔显示在同一横幅中"，但邮件部分未定义对应的合并规则。若管理员同一天添加了 KeyBanc 2026 和 High Alpha 2026 两个新 Edition，用户是收到 2 封独立邮件，还是 1 封合并邮件？触发 3 次新平台添加时同理。邮件是否有频率限制或聚合窗口？
- **建议**：明确 2.1 工作流邮件的合并策略：是实时各自独立发送，还是设定聚合窗口（如同一天内合并为一封）；若合并，说明邮件正文中多个 platform-Edition 的展示格式。

**[R-04] 🔴 触发阈值描述与示例存在矛盾**
- **位置**：2.2 节触发条件第 2 点
- **维度**：明确性、可测试性
- **问题**：文档描述为"百分位变化**超过** 10（如由 P10 变为 P20）"，但 P10→P20 的变化值恰好等于 10，属于"等于 10"而非"超过 10"。如果阈值为严格大于 10，则该示例是错误的；如果示例是正确的（即包含等于 10 的情况），则触发条件应改为"大于等于 10"。两者判断代码不同，测试用例边界不同。
- **建议**：明确阈值是"≥ 10"还是"> 10"，并确保描述与示例一致。

**[R-05] 🟡 "首次登录"的作用域定义不明确**
- **位置**：2.1 节横幅提示第 3 点、2.2 节横幅提示第 2 点
- **维度**：明确性
- **问题**："首次登录会提示"是指触发事件发生后该用户首次登录时展示横幅，还是仅限于该用户注册后的第一次登录？语义上存在歧义。从业务语境推测应为前者，但需明确。
- **建议**：改为"触发事件发生后，用户首次登录时展示横幅"，消除歧义。

**[R-06] 🟡 超链接权限异常处理逻辑不完整**
- **位置**：2.1 节超链接跳转异常处理、2.2 节超链接跳转异常处理
- **维度**：完整性
- **问题**：文档仅说明"若 portfolio portal 人员不再有对应公司的访问权限，则页面横幅不显示"，但未说明该情况下邮件是否仍然发出。若邮件已发出，用户点击超链接后进入系统但页面无横幅，用户体验存在割裂；若邮件也应不发，则需要在邮件发送逻辑中增加实时权限校验，这是额外的技术要求。
- **建议**：补充说明：当用户已失去访问权限时，邮件通知是否同样跳过该用户，还是仍发送但横幅不展示；并说明权限校验的时机（邮件发送时、用户点击时或页面加载时）。

**[R-07] 🟡 2.2 邮件中超链接跳转角色名称存在疑似笔误**
- **位置**：2.2 节邮件提示超链接说明
- **维度**：一致性
- **问题**：2.2 节邮件超链接描述为"portfolio **portfolio** 角色"，而 2.1 节相同位置写的是"portfolio **portal** 角色"，前者疑为笔误，但若为刻意区分则需说明两者的差异。
- **建议**：确认是否为"portfolio portal 角色"的笔误，统一术语后修正。

**[R-08] 🟡 "closed month" 未定义，直接影响 2.2 触发逻辑理解**
- **位置**：2.2 节触发条件
- **维度**：明确性
- **问题**：触发条件中三处均使用"closed month 月份"这一术语，但文档未解释其含义（是账期关账月？财年关闭月？还是系统中某个状态字段的枚举值？），开发人员无法准确实现监测范围的边界判断。
- **建议**：在触发条件前或术语表中定义"closed month"的含义及其在系统数据模型中的对应字段。

**[R-09] 🟡 2.2 同一公司多条触发条件同时满足时的通知策略未定义**
- **位置**：2.2 节触发条件
- **维度**：完整性
- **问题**：2.2 包含 3 个独立触发条件，若某公司在同一天同时满足条件 2 和条件 3（指标值无变化的百分位漂移 + 指标值变化导致的百分位漂移），系统是否发送 2 封邮件、2 个横幅？还是合并为 1 条通知？未定义合并策略会导致用户在同一天收到重复内容的通知。
- **建议**：补充说明 2.2 中同一公司在同一日检测周期内触发多条条件时的通知合并规则。

**[R-10] 🟡 2.1 与 2.2 横幅通知接收角色范围不一致，未说明是否为刻意设计**
- **位置**：2.1 节横幅提示、2.2 节横幅提示
- **维度**：一致性
- **问题**：2.1 横幅仅对"Portfolio manager"和"Company Admin"显示；2.2 横幅对"Portfolio manager 或**其他有该公司权限的人**"以及"Company Admin"显示。两个工作流通知范围不同，文档未说明这一差异是出于业务逻辑考量（如行业基准更新影响面更窄）还是遗漏。
- **建议**：明确两个工作流横幅接收范围不一致的业务原因，或统一为相同范围并注明。

**[R-11] 🟡 特殊情况说明的适用范围不明确**
- **位置**：特殊情况说明章节
- **维度**：明确性
- **问题**：文档末尾说明"公司状态为 Exited 和 Shut down 的，未绑定 portfolio 的不进行监测"，但未明确此规则仅适用于 2.2（内部基准监测）还是同时适用于 2.1（行业基准更新触发的通知）。由于 2.1 是由管理员操作触发、非逐公司监测，其适用性本身存在歧义。
- **建议**：补充说明该特殊情况的适用范围（仅限 2.2，或同时约束 2.1 的通知对象筛选）。

**[R-12] 🔵 2.2 每日监测的性能边界未提及**
- **位置**：2.2 节监测频率
- **维度**：可行性
- **问题**：2.2 需每日对所有满足条件的公司监测 6 个指标的百分位变化，若监测公司数量较多，任务执行时长和数据库查询压力需要评估，文档未提及是否有时间窗口限制或分批处理策略。
- **建议**：评估监测公司数量级，说明任务是否需要分批执行，并在非功能性需求中加入执行时间窗口约束。

**[R-13] 🔵 通知记录/历史日志需求未提及**
- **位置**：全文
- **维度**：完整性
- **问题**：文档未提及系统是否需要记录已发送的通知历史（如通知时间、接收人、触发原因），缺少审计支持，也无法支持后续"用户是否已收到通知"的排查需求。
- **建议**：明确是否需要持久化通知发送记录，如需要，描述最低记录字段（触发类型、触发时间、接收人、发送结果）。

**[R-14] 🔵 文档中存在文字笔误影响阅读**
- **位置**：2.1 节邮件提示第 1 点
- **维度**：明确性
- **问题**："满足出**发**条件后"应为"满足触**发**条件后"，属于明显笔误。
- **建议**：修正为"触发条件"。
