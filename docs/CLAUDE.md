# CLAUDE.md — docs 目录规范

## 目录结构

```
docs/
├── CLAUDE.md                         ← 本文件（创建规则）
├── <功能名称>/                        ← 每个功能一个目录
│   │
│   ├── 调研/                         ← 第一阶段：调研
│   │   ├── design-philosophy.md          设计理念（WHY）
│   │   ├── system-architecture.md        系统架构（HOW 系统层）
│   │   ├── java-design.md                Java 端设计
│   │   ├── python-design.md              Python 端设计
│   │   ├── frontend-design.md            前端设计
│   │   └── code-examples.md              代码示例
│   │
│   ├── 需求/                         ← 第二阶段：需求
│   │   └── requirement-doc.md            需求文档（PRD）
│   │
│   ├── 需求审核/                     ← 第三阶段：需求审核
│   │   └── requirement-review.md         需求审核报告
│   │
│   ├── 设计/                         ← 第四阶段：设计
│   │   └── design-doc.md                 设计文档（功能设计、接口设计、数据模型）
│   │
│   ├── 开发设计/                     ← 第五阶段：开发设计
│   │   ├── dev-design-doc.md             开发设计文档（面向开发的详细实现方案）
│   │   └── code-examples.md              实现参考代码
│   │
│   ├── 开发设计审核/                 ← 第六阶段：开发设计审核
│   │   └── dev-design-review.md          开发设计审核报告
│   │
│   ├── 测试用例/                     ← 第七阶段：测试用例
│   │   └── test-cases.md                 手动测试用例（功能/边界/UI/验收）
│   │
│   ├── 测试用例审核/                 ← 第八阶段：测试用例审核
│   │   └── test-cases-review.md          测试用例审核报告
│   │
│   └── 单元测试/                     ← 第九阶段：单元测试
│       ├── unit-test-doc.md              单元测试文档（测试策略、覆盖率要求）
│       └── README.md                     测试代码索引（指向实际测试文件位置）
│
└── <另一个功能>/
```

## 文档流水线（9 阶段）

```
调研 → 需求 → 需求审核 → 设计 → 开发设计 → 开发设计审核 → 测试用例 → 测试用例审核 → 单元测试
 ①      ②       ③        ④       ⑤          ⑥             ⑦           ⑧            ⑨
```

每个阶段对应一个子目录，按顺序产出：

| 阶段 | 子目录 | 核心产出 | 输入依赖 |
|------|--------|----------|----------|
| ① 调研 | `调研/` | 设计理念、系统架构、各端设计 | Asana EPIC / 原始需求 |
| ② 需求 | `需求/` | 需求文档 (PRD) | 调研产出 |
| ③ 需求审核 | `需求审核/` | 审核报告（通过/不通过 + 问题清单） | 需求文档 |
| ④ 设计 | `设计/` | 功能设计文档（接口、数据模型、交互） | 审核通过的需求 |
| ⑤ 开发设计 | `开发设计/` | 开发设计文档（实现方案、代码结构、技术细节） | 设计文档 |
| ⑥ 开发设计审核 | `开发设计审核/` | 审核报告（架构合理性、安全、性能） | 开发设计文档 |
| ⑦ 测试用例 | `测试用例/` | 手动测试用例（功能/边界/UI/验收） | 需求 + 设计文档 |
| ⑧ 测试用例审核 | `测试用例审核/` | 审核报告（覆盖率、遗漏场景） | 测试用例 |
| ⑨ 单元测试 | `单元测试/` | 单元测试文档 + 测试代码索引 | 开发设计 + 测试用例 |

## 各阶段文档内容规范

### ① 调研文档

调研阶段按职责拆分为多个文档，**一个文档一个关注点**：

| 文档 | 关注点 | 读者 |
|------|--------|------|
| `design-philosophy.md` | 为什么这样做（原则、trade-off） | 全团队 |
| `system-architecture.md` | 系统层：数据流、SQS、API、安全、分期 | 架构师 + 全栈 |
| `java-design.md` | Java 端：模块结构、表设计、SQS 集成 | Java 开发 |
| `python-design.md` | Python 端：AI 引擎、映射、记忆、提示词 | Python 开发 |
| `frontend-design.md` | 前端：路由、组件、dva、交互、轮询 | 前端开发 |
| `code-examples.md` | DDL + 参考代码（Java/Python/SQL 分区） | 全栈 |

### ② 需求文档

- 纯业务需求，**禁止包含技术实现细节**
- 包含：功能描述、业务规则、公式定义、用户角色、验收标准
- 文件：`requirement-doc.md`

### ③ 需求审核

- 对照需求文档逐项审查完整性、一致性、可行性
- 输出：通过/不通过 + 按严重级别分类的问题清单
- 文件：`requirement-review.md`

### ④ 设计文档

- 功能设计（非代码级别）：接口定义、数据模型、页面交互、状态流转
- 面向 PM 和开发共同理解的中间层
- 文件：`design-doc.md`

### ⑤ 开发设计文档

- 面向开发的详细实现方案：代码结构、模块划分、关键算法、异常处理
- 按端拆分（如需要可创建 `dev-java.md`、`dev-python.md`、`dev-frontend.md`）
- 文件：`dev-design-doc.md` + 可选 `code-examples.md`

### ⑥ 开发设计审核

- 审查开发设计的架构合理性、安全性、性能、与需求的覆盖度
- 使用多角色审核（架构师 + 安全 + 领域专家）
- 文件：`dev-design-review.md`

### ⑦ 测试用例

- 手动测试用例：功能测试、边界测试、UI 测试、验收测试
- 每条用例包含：前置条件、操作步骤、预期结果
- 文件：`test-cases.md`

### ⑧ 测试用例审核

- 审查测试用例的覆盖率、是否遗漏边界场景、是否与需求对齐
- 文件：`test-cases-review.md`

### ⑨ 单元测试

- 单元测试策略：覆盖率目标（≥80%）、测试框架、Mock 策略
- 测试代码不放在 docs 中，放在各子项目的 test 目录，此处只做索引
- 文件：`unit-test-doc.md` + `README.md`（指向测试代码位置）

## 文件命名规则

- 使用 **英文小写 + 连字符**：`system-architecture.md`，不用中文文件名
- 功能目录名可以用中文：`智能解析/`、`Normalization Tracing/`
- 子目录名固定中文：`调研/`、`需求/`、`需求审核/`、`设计/`、`开发设计/`、`开发设计审核/`、`测试用例/`、`测试用例审核/`、`单元测试/`

## 创建新功能文档时

1. 在 `docs/` 下创建功能目录：`docs/<功能名>/`
2. 按当前所处阶段创建对应子目录
3. 按上述模板创建文档，不需要的文档不创建
4. 每份文档头部必须有 `> 关联文档:` 交叉引用行
5. 阶段可以跳过（如无需调研可直接从需求开始），但顺序不能颠倒

## 禁止事项

- 不要在一个文件中混写 Java + Python + 前端的实现细节
- 不要把代码示例（DDL、完整函数）写在设计文档中，统一放 `code-examples.md`
- 不要使用绝对路径引用文件，用相对路径：`[系统架构](./system-architecture.md)`
- 不要创建 `technical-design.md` 这种万能文档 — 按职责拆分
- 不要在 docs 中放测试代码文件 — 测试代码在各子项目 test 目录
