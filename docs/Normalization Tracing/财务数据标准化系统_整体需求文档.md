# 财务数据标准化系统 - 整体需求文档

## 一、功能概述

### 1.1 系统定位
**Normalize Financials** 是一个财务数据标准化平台，支持 Looking Glass 将多源异构财务输入（QuickBooks、Financial Entry、手动上传）转换为标准化、符合GAAP（General Accepted Accounting Principle）的统一结构，同时支持预测数据（承诺预测和系统生成预测）的标准化处理。

### 1.2 支持的数据类型
- **Historical Actuals（实际财务数据）**：历史财务记录
- **Committed Forecast（承诺预测）**：用户手动提交的财务预测
- **System Generated Forecast（系统生成预测）**：算法引擎自动生成的预测

---

## 二、核心业务逻辑

### 2.1 规范化引擎（Normalization Engine）

#### 2.1.1 工作原理
系统采用 **通用核心逻辑 + 规则集配置** 的设计：
- 核心引擎逻辑在所有数据类型间统一共享
- 每个公司分配一个专属的规范化规则集（Ruleset）
- 规则集定义公司特定的配置项

#### 2.1.2 规则集配置项
规则集必须定义：
1. **货币转换** - 使用哪个月度汇率进行转换
2. **收入展示方法** - 支持三种方式：
   - Last Month Method：ARR = 最近一个月MRR × 12
   - 3-Month Average Method：ARR = 最近3个月MRR平均值 × 12
   - Trailing Twelve Months (TTM)：ARR = 过去12个月收入总和
3. **会计基础属性** - 现金制或应计制（仅作为描述属性，不做转换）

**MVP不包含**：财政日期对齐规则、会计基础转换逻辑（架构预留扩展空间）

---

### 2.2 数据映射逻辑（Mapping Logic）

#### 2.2.1 映射规则
系统基于规则将原始指标行项目映射到标准化类别：

**标准映射关系表**：
| 原始数据（LG） | 目标类别 |
|---|---|
| Gross Revenue | Revenue |
| COGS | COGS |
| S&M Expenses | OPEX |
| S&M Payroll | OPEX |
| R&D Expenses | OPEX |
| R&D Payroll | OPEX |
| G&A Expenses | OPEX |
| G&A Payroll | OPEX |
| Misc OpEx | OPEX |
| 其他行项目 | 各自独立行项目 |

#### 2.2.2 映射存储与追踪
每条标准化记录必须存储：
- 原始源记录 ID
- 原始指标名称与金额
- 分配的标准化类别
- 使用的映射规则或逻辑
- 映射版本标识符

#### 2.2.3 映射版本管理
- 映射逻辑版本化管理
- 映射规则变更时：
  - 新的规范化运行使用更新的映射逻辑
  - 历史标准化数据保持不变
  - 系统支持使用历史版本规则重新规范化

---

### 2.3 会计方法设置（Cash/Accrual Setting）

#### 2.3.1 配置说明
每个公司在 **Company Settings > Finance 标签页** 必须有一项会计方法设置：
- 选项：Cash 或 Accrual（默认）
- 存储位置：公司级别
- 属性：纯元数据，不作为转换规则

#### 2.3.2 规范化行为
- 规范化引擎 **不执行** 现金/应计转换
- 规范化 **不改变** 基于会计方法的财务数值
- 会计方法仅用于下游用途（Benchmark中的同类分组）
- 会计方法记录为规范化运行元数据（审计可追踪）

#### 2.3.3 基准测试与变更管理
- 基准测试逻辑使用会计方法进行同类分组（现金公司仅与现金公司对标）
- 修改会计方法 **不影响** 历史标准化数据
- 修改会计方法 **不触发** 历史数据重新规范化
- 修改仅影响未来规范化和基准测试

---

### 2.4 历史数据规范化（Historical Normalization Stored Procedure）

#### 2.4.1 触发机制
自动每晚执行，检测自上次成功规范化运行以来的原始财务数据变化：
- 手动数据输入
- QuickBooks 同步更新
- 先前提交财务的修正

#### 2.4.2 数据稳定性判断
规范化仅在数据稳定/关闭时执行：
- **QB公司**：每月15日之后
- **手动输入公司**：数据输入/更新后立即执行

#### 2.4.3 处理流程
1. 识别有新增或更新原始数据的公司
2. 仅对检测到变化的公司执行规范化（无变化则跳过）
3. 对每个符合条件的公司：
   - 应用当前规范化规则集
   - 使用公司的财务设置（财年开始月、会计方法、货币处理）
   - 生成完整的月度标准化数据集

#### 2.4.4 版本管理与数据保留
- 每次成功规范化创建新版本
- 历史标准化数据集保持不变且可访问
- **永不覆盖或删除** 规范化数据
- 每条规范化记录关联：
  - 规范化版本
  - 规则集版本
  - 执行时间戳

#### 2.4.5 错误处理
- 一个公司的失败 **不阻止** 其他公司处理
- 公司规范化无法完成时：
  - 记录清晰的失败原因
  - 之前的规范化数据集保持活跃
- 部分数据问题（缺汇率等）：
  - 带警告完成处理
  - 记录验证消息

#### 2.4.6 日志与可见性
每次夜间运行记录：
- 处理的公司清单
- 跳过的公司清单
- 每个公司状态：Success / Success with Warnings / Failed
- 验证警告和错误，关联到规范化版本
- 每个公司最近成功规范化可识别

#### 2.4.7 下游消费
规范化输出立即可用于：
- 基准测试
- 投资组合分析
- 无需额外转换

---

### 2.5 预测数据标准化处理（Forecast Data Normalization）

#### 2.5.1 核心需求
扩展标准化引擎支持预测数据处理，使用相同的规则引擎处理：
- **承诺预测（Committed Forecast）**：用户提交的财务预测
- **系统生成预测（System-Generated Forecast）**：算法引擎自动生成的预测

#### 2.5.2 处理原则
- 复用现有标准化引擎（不创建重复逻辑）
- 预测数据与历史数据使用相同的转换规则
- 应用所有相同的转换：货币转换、收入展示标准化、衍生指标计算
- 预测结果独立存储（与历史表分离，不相互影响）

#### 2.5.3 重新处理触发机制

| 触发条件 | 处理方式 |
|---------|---------|
| 承诺预测被编辑或接受导致的数据变化 | 自动重新处理 |
| 系统生成预测被刷新导致的数据变化 | 自动重新处理 |
| 公司规则集变更导致的数据变化 | 自动重新处理所有相关预测 |
| 汇率更新导致的数据变化| 自动重新处理相关预测 |


#### 2.5.4 版本管理
每次预测数据标准化运行生成唯一版本，版本信息包含：
- 公司ID
- 预测类型（Committed或System-Generated）
- 应用的规则集版本
- 处理时间戳
- 财政年度

#### 2.5.5 基准对标支持
- 标准化预测指标持久化存储
- 预测指标可直接用于预测基准对标计算
- 支持预测与历史的基准对标比较

### 2.6 规范化溯源（Normalization Tracing）

#### 2.6.1 访问权限
- **仅admin portal用户** 可访问


#### 2.6.2 追踪流程（分级钻取）

用户按以下步骤进行追踪：

| 步骤 | 操作名称 | 用户操作 | 系统反馈 |
|------|---------|---------|--------|
| 1 | 选择公司 | Admin 选择目标公司 | 确认选择 |
| 2 | 选择财务期间 | 选择特定月份 | 确认选择 |
| 3 | 选择数据类型 | 在三种类型中选择 | 显示对应数据可用性 |
| 4 | 选择标准化指标/类别 | 选择要追踪的指标 | 显示追踪结果 |

**数据类型选择项**：
- Historical Actuals（实际数据）
- Committed Forecast（承诺预测）
- System Generated Forecast（系统生成预测）

#### 2.6.3 源系统确定逻辑
源系统在 **公司-月份级别** 确定，而非每个指标单独确定。

根据摄入历史识别唯一权威源系统：
- **仅手动输入** → 所有月份源系统 = Manual Entry
- **仅QB** → 所有月份源系统 = QuickBooks API
- **QB → 手动切换** → 所有月份源系统 = Manual Entry
- **手动 → QB 切换（启用覆盖）** → 所有月份源系统 = QuickBooks API
- **手动 → QB 切换（未启用覆盖）**：
  - 历史月份 + 当前月份 → Manual Entry
  - 未来月份 → QuickBooks API

#### 2.6.4 Metric Detail展示

**页面分为三个清晰分区**：

**1. 标准化输出**
- 数据类型标识（Historical Actuals / Committed Forecast / System Generated Forecast）
- 标准化指标名称
- 标准化数值（USD）
- 所选月份的已解析源系统（突出显示）
  - 示例："Source System (for selected month): Manual Entry"

**2. 原始源数据**
- 原始指标/类别名称
  - 若非1对1关系，显示 "Aggregated from multiple sources"
- 原始金额
  - 若非1对1关系，显示 "-"
- 源系统（继承自月份级别确定）
- 按升序排列（主排序键：Company，次排序键：Metric Name）
- "More Details" 按钮：查看完整信息

**3. 规范化与计算逻辑**
- **科目映射**："Mapped to Normalized Category"
- **货币转换详情**：
  - 原始货币
  - USD数值
  - 使用的汇率
  - 转换日期
- **衍生计算逻辑**：
  - 显示公式以追踪标准化输出来源
  - 信息提示：说明指标来自规范化表或LG FE表
  - 多个公式时在弹窗展示

#### 2.6.5 数据特性
- 所有数据为 **只读**
- 数据反映所选周期的 **当前活跃规范化版本**
- 溯源输出与下游基准测试和分析使用的数据 **保持一致**

## 三、操作流程

### 3.1 用户追踪操作流程

```
Admin 用户访问 Normalization Tracing 页面
    ↓
选择公司
    ↓
选择财务期间（月份）
    ↓
【关键】选择数据类型
  ├─ Historical Actuals
  ├─ Committed Forecast
  └─ System Generated Forecast
    ↓
系统识别数据类型，加载对应的活跃规范化版本
    ↓
显示可用的标准化指标/类别列表
    ↓
选择要追踪的指标
    ↓
系统查询对应的源数据和转换规则
    ↓
展示完整追踪链路：
  ├─ 标准化输出值
  ├─ 原始源数据
  ├─ 映射规则
  ├─ 货币转换详情
  └─ 衍生计算公式
```

## 四、操作反馈

### 4.1 追踪操作的反馈

| 操作 | 即时反馈 | 延迟/后续反馈 |
|------|--------|------------|
| 选择公司 | 期间列表刷新 | — |

### 4.2 系统状态反馈

| 场景 | 反馈形式 | 内容 |
|------|--------|------|

## 五、数据模型

### 5.1 数据粒度与版本
- **时间粒度**：月度（Monthly Granularity）
- **版本管理**：每次规范化创建新版本，历史数据保持不变
- **数据状态**：Raw → Mapped → Normalized → Approved
- **追踪信息**：规范化版本、规则集版本、执行时间戳、数据类型

### 5.2数据存储隔离
- **历史标准化表**：存储 Historical Actuals 的规范化结果
- **预测标准化表（Committed）**：存储 Committed Forecast 的规范化结果
- **预测标准化表（System-Generated）**：存储 System Generated Forecast 的规范化结果
- 三个表独立维护，互不影响

## 六、数据来源与存储

### 6.1 输入数据来源

| 数据项 | 来源系统 | 数据类型 | 更新频率 |
|--------|--------|---------|--------|
| 实际财务数据 | QuickBooks API / Financial Entry / Upload+OCR | Historical | 月度或实时 |
| 承诺预测数据 | 用户在平台手动输入 | Committed Forecast | 用户维护 |
| 系统生成预测数据 | 系统预测模型 | System Generated Forecast | 模型定期生成 |
| 标准化规则 | 配置管理系统 | 规则集 | 管理员更新 |
| 货币汇率 | 外部数据源 / 手动维护 | 汇率数据 | 日度或月度 |
| 追踪审计日志 | 系统生成 | 审计日志 | 实时记录 |
| 会计方法设置 | 公司配置 | 元数据 | 管理员维护 |
