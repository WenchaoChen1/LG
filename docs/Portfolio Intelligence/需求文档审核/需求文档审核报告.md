### 2. **Filter 表定义与实际不符**
**位置**：第128-135行（表）和第217-223行（重复表）  
**问题**：
- GROWTH → 包含的是"ARR Growth Rate" ✓
- EFFICIENCY → 包含的是"Gross Margin"，但通常 EFFICIENCY 应包含多个指标
- MARGINS → 包含的是"Monthly Net Burn Rate, Monthly Runway"，但这应该属于"Burn & Runway"
- CAPITAL → "Rule of 40, Sales Efficiency Ratio" ✓
- 表格重复定义两次，浪费空间，且没有说明逻辑

**修改建议**：
```
重新整理 Filter 的含义，确认产品设计意图：
- GROWTH：收入增长相关  → ARR Growth Rate
- EFFICIENCY：运营效率相关 → Gross Margin、Sales Efficiency Ratio（调整）
- MARGINS：利润空间相关  → Gross Margin（可能重复）
- CAPITAL：资本效率相关  → Rule of 40、Sales Efficiency Ratio
- BURN：烧钱能力  → Monthly Net Burn Rate、Monthly Runway（新增）

或者确认 MARGINS 和 EFFICIENCY 的区别。
建议与产品确认，然后删除表格重复。
```

### 3. **Overall Benchmark Score 计算逻辑缺失**
**位置**：第207行  
**问题**：
文本只说"数据取值为后面此行中所有百分位的算术平均值"，但没有说明：
- 当选择多个 Benchmark 时，如何聚合？
- 当选择多个 Data 时，如何聚合？
- "百分位是 N/A 时不参与计算"——但什么情况下会是 N/A？

**修改建议**：
```
补充完整的计算逻辑：

Overall Score = Mean(当前行所有有效的百分位数)

其中"有效的百分位"指：
- 在用户选择的 Data 类型范围内
- 在用户选择的 Benchmark 范围内
- 该指标该公司该基准该数据类型有数据（不是 N/A）

示例：
- 若选 Data = [Actuals, Committed Forecast]、Benchmark = [Internal Peers]、Filter = All
- 则 Overall Score = Mean(Actuals-Internal Peers 的6个指标百分位 + Committed Forecast-Internal Peers 的6个指标百分位)
  = Mean(最多12个百分位，排除N/A)
```

### 4. **Trend 视图数据缺失处理相悖**
**位置**：第348行  
**问题**：
文本说"某家公司某个月份无数据，则展示最低点 P0"  
但在 Snapshot 视图等其他地方，数据缺失应该显示 "N/A"  
两种处理方式相悖，用户会困惑

**修改建议**：
```
统一数据缺失处理逻辑：

方案A（推荐）：
- 数据缺失时，不显示数据点，折线断开或显示缺口
- 不要人为赋值 P0，这会造成误导

方案B：
- 数据缺失时，显示 "N/A" 或特殊符号
- Trend 视图中该公司在该月份的点位不显示，但不影响其他月份

选择其中一个，删除"展示最低点P0"的说法。
```

### 5. **同行数据有效性判断逻辑不清**
**位置**：第266-268行  
**问题**：
- "从 closed month 开始往前数 24 个日历月内，连续六个月非负 gross revenue"
- 这里"连续六个月"的条件很严格，但是否会导致大多数公司无有效同行？
- 没有说明"有效同行 < 3 家"时的降级方案（提示说是"回退全平台基准"，但无 Fallback 提示案例）

**修改建议**：
```
补充说明：
1. 数据有效性示例：
   例如，当前是2026年4月（Actuals数据到2026年3月），往前数24个月到2024年4月。
   在这24个月内，需要找到"连续6个月"非负gross revenue的公司。
   什么是"连续"？是否允许缺口？如何处理新公司？

2. Fallback 机制：
   - 若有效同行 < 3家，回退到全平台
   - UI 应显示："No direct peers found. Showing all active companies."
   - 此时基准值应该如何显示？（当前未说明）

3. 考虑降级处理：
   - 若 < 3 家，是否可考虑扩大范围（如放宽阶段匹配）而非直接全平台？
```

### 8. **百分位外范围处理规则过于复杂且易出错**
**位置**：第291-293行  
**问题**：
"大于 P25，则展示 > P25，计算时用 P50"这样的规则看起来像是临时补救方案，会导致：
- 数据计算不准确
- 代码复杂度高，易出 bug
- 用户看到"> P25"和计算用 P50 的心理不一致

**修改建议**：
```
重新审视设计：
1. 是否一开始就应该要求外部基准数据完整（至少P25、P50、P75）？
2. 还是允许不完整，但有明确的、一致的插值策略？

建议采用明确的降级策略：
- 若只有P50：
  - value > P50 → 估算为 P75（显示~P75）
  - value < P50 → 估算为 P25（显示~P25）
  - value = P50 → 显示 P50

不要创建"大于P25用P50"这样的中间值转换逻辑。
```
