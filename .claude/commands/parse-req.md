# 解析并补全需求文档

读取 Word 或任意格式的原始需求文档，分析并补全为完整闭环的需求规格，保存到**与源文件同级目录**，供后续生成开发 skill 使用。

## 使用方式

```
/parse-req <文档路径>
```

例如：
- `/parse-req requirements/Looking Glass.docx`
- `/parse-req requirements/raw/用户管理.txt`

---

## 执行步骤

### 第一步：读取原始文档

根据 `$ARGUMENTS` 指定的文件扩展名选择读取方式：

- **`.docx` 文件**：使用以下 Bash 命令提取全文内容：
  ```bash
  python -c "
  import sys
  sys.stdout.reconfigure(encoding='utf-8')
  from docx import Document
  doc = Document('<文件路径>')
  for para in doc.paragraphs:
      if para.text.strip():
          print(para.text)
  "
  ```
- **`.txt` / `.md` 文件**：直接用 Read 工具读取
- **其他格式**：提示用户转换为 `.txt` 或 `.md`，或直接粘贴内容

提取完成后，标记**哪些章节缺失或不完整**。

### 第二步：结合项目上下文补全需求


按以下维度逐一检查并补全，**缺失的必须推断补全，不得留空**：

| 维度 | 检查项 |
|------|--------|
| 业务目标 | 背景、目的、成功指标 |
| 用户故事 | 角色、操作、价值 |
| 功能需求 | 核心流程、分支流程、边界条件 |
| 异常处理 | 错误场景、提示文案、降级策略 |
| API 设计 | 新增/修改接口、请求/响应结构 |
| 数据模型 | 表结构变更、字段说明、索引 |
| 前端交互 | 页面描述、组件、交互逻辑、字段校验 |
| 权限控制 | 涉及角色、role 权限码、显隐逻辑 |
| 非功能需求 | 性能、安全、兼容性 |
| 测试场景 | 正常流程用例、异常流程用例 |
| 验收标准 | 可量化的完成判断标准 |
| 依赖与影响 | 依赖模块、受影响模块、风险 |

### 第三步：展示补全差异

在保存前，用以下格式展示原文档与补全内容的对比：

```
原文档已包含：✓ 业务背景、✓ 核心功能描述
AI 补全内容：+ API设计、+ 数据模型、+ 权限控制、+ 测试场景、+ 验收标准
推断依据：<说明依据来源>
```

询问用户：**确认保存 / 修改某项 / 重新生成**

### 第四步：章节完整性检查

保存前检查所有 11 个章节（第 1-11 章）是否都有实质内容：
- 若存在空章节或内容仅为占位符（如"待补充"、"<描述>"），列出这些章节名称并提示用户补全或由 Claude 继续推断
- 所有章节确认有实质内容后，方可进入第五步

### 第五步：保存完整需求文件

**保存路径**：`features/<功能名称>/req-spec.md`
（若目录不存在先创建 `features/<功能名称>/` 文件夹）

功能名称从源文件名推断（去掉路径和扩展名，转为小写连字符格式）。

例如：
- 源文件：`requirements/raw/Looking Glass.docx` → 保存为：`features/looking-glass/req-spec.md`
- 源文件：`requirements/raw/用户管理.txt` → 保存为：`features/用户管理/req-spec.md`
