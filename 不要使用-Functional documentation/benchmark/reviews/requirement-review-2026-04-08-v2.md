### 功能理解

Benchmark 是 LG 财务模块中的对标分析功能。核心逻辑：用户选定月份、数据类型（Actuals/Committed Forecast/System Generated Forecast）和基准来源（Internal Peers / 三家外部平台），系统计算 6 个关键财务指标的百分位排名。指标按 4 个板块（Revenue & Growth、Profitability & Efficiency、Burn & Runway、Capital Efficiency）分组，每个板块汇总出板块分数，4 个板块再汇总出 Overall Benchmark Score。页面有两种视图：Snapshot（单月快照）展示分布条和百分位点位，Trend（趋势）展示折线图，底部还有一个六维雷达图（Metrics Summary）汇总全貌。

DATA x BENCHMARK 组合最多产生 12 个"维度"，每个维度独立计算百分位。Internal Peers 使用 Nearest Rank Percentile 公式 `P = (R-1)/(N-1) * 100%`；外部基准使用 P25/P50/P75 锚点加线性插值。同行匹配需满足公司类型、阶段、会计方法、ARR 范围四维完全一致，不足 4 家则回退到 LG 全平台活跃公司。

### 🔴 阻断性问题（必须修复才能进入设计）

> 逻辑矛盾、规则缺失、公式错误、关键信息丢失——不修复则开发无法动工

#### 1. FILTER=ALL 的交互行为在两处描述完全矛盾

- **位置**：§3.1 第 49 行 vs §4.1.2 第 103 行
- **原文**：
  - 第 49 行：「ALL与其他选项互斥，选中ALL后如果选择其他选项，ALL就自动取消勾选，其他选项可同时选中」
  - 第 103 行：「ALL选中时其他自动选中，取消ALL时其他自动清空」
- **问题**：这两段描述了截然不同的行为模式。第 49 行说"ALL 与其他互斥"——选 ALL 后再选其他，ALL 消失；第 103 行说"选 ALL 联动全选其他"——ALL 是全选快捷键。开发不知道该信哪个。这是整个筛选交互的基础逻辑，矛盾会导致前端实现方向完全不同。
- **建议**：统一为一种行为，并在 §4.1.2 的表格中明确描述以下全部场景：
  1. 用户点击 ALL → 其他选项是否联动选中？
  2. ALL 已选中 → 用户取消某一个选项 → ALL 是否自动取消？
  3. ALL 未选中 → 用户逐一选完全部选项 → ALL 是否自动选中？
  4. 用户取消 ALL → 其他选项是否全部取消？

#### 2. 所有指标值完全相同时的处理在两处描述完全矛盾

- **位置**：§6.1.1 第 412 行 vs §7.4 第 567-569 行
- **原文**：
  - 第 412 行：「若所有指标值完全相同：不显示百分位，不计入 metric summary 和 Overall Score」
  - 第 567-569 行：「分配相同的百分位值（如都是 50%ile）」「正常参与权重计算」
- **问题**：一处说"不显示、不计入"，另一处说"正常显示、正常计入"。开发完全无法判断哪个正确。此外，§6.1.1 的描述是"所有指标值完全相同"，§7.4 是"多个公司的所有指标值都完全相同"，主语也不一致——到底是"某一个指标所有公司值相同"还是"所有指标所有公司值都相同"？
- **建议**：明确两个独立场景：（1）某单个指标所有公司值相同 →如何处理；（2）所有指标所有公司值都相同 → 如何处理。删除矛盾的那一处。

#### 3. 百分位超范围的汇总计算值在两处矛盾

- **位置**：§6.1.2 第 434 行 vs §7.5 第 575-576 行
- **原文**：
  - 第 434 行：「若大于P75，展示＞P75，计算时用P100；若小于P25，展示＜P25，计算时用P0」
  - 第 575-576 行：「高于最高值：显示 >P75，汇总计算使用 P75」「低于最低值：显示 <P25，汇总计算使用 P25」
- **问题**：>P75 在汇总时到底用 P100 还是 P75？<P25 用 P0 还是 P25？两处给出相反答案。此外，第 434 行还列出了 >P25 用 P50、>P50 用 P75、<P50 用 P25、<P75 用 P50 等规则，但没有解释这些中间情况何时会出现（三个锚点 P25/P50/P75 都有数据时，>P25 并不算"超范围"）。怀疑第 434 行描述的是"只提供部分锚点"的场景，但文档未明确说明前提条件。
- **建议**：
  1. 统一超范围汇总值（P100/P0 还是 P75/P25）
  2. 明确第 434 行各规则的适用前提（是否只在部分锚点缺失时生效）
  3. 用一张完整表格列出：可用锚点组合 × 值位置 → 显示文本 + 汇总计算值

#### 4. Monthly Net Burn Rate 指标方向与排序描述自相矛盾

- **位置**：§5.1.3 第 335-336 行
- **原文**：
  - 「方向：正向指标」
  - 「排序方式：正向排序（按数值大小从小到大排序，数值小的排名靠前，百分位更高）」
- **问题**："正向指标"的通常含义是"数值越高越好"。但描述紧接着说"数值小的百分位更高"，即"数值越低越好"，这与"正向指标"直接矛盾。结合百分位公式 `P = (R-1)/(N-1)`，升序排列下 R=1（最小值）得 P=0%，R=N（最大值）得 P=100%——即"大值高百分位"。所以"数值小的百分位更高"在数学上也与公式矛盾。
  
  从业务含义看，Monthly Net Burn Rate = Net Income - Capitalized R&D，值越大（亏损越少或盈利越多）确实越好，应属正向指标 + 升序排列 + 大值高百分位。因此"数值小的排名靠前，百分位更高"这句话是错的。
- **建议**：删除"数值小的排名靠前，百分位更高"这句话，改为「升序排列，数值越大百分位越高，与正向指标一致」。同时确认这一排序逻辑与 Sales Efficiency Ratio（反向指标）的排序方向形成对比说明。

#### 5. Overall Score 在多维度选择下如何确定单一数值未说明

- **位置**：§4.1.4 第 124-126 行 + §6.4.2 第 498-504 行
- **原文**：
  - 「计算方法：四个板块分数的算术平均值」
  - 「板块分数 = 该板块所有有效指标百分位的算术平均」
- **问题**：当用户同时选择了多个 DATA（如 Actuals + Committed Forecast）和多个 BENCHMARK（如 Internal Peers + KeyBanc）时，每个指标会产生 4 个不同百分位（2x2 维度组合）。板块分数用哪个维度的百分位？Overall Score 卡上展示的百分位数值（如截图中的"~67P"）取自哪个维度？

  §6.4.3 定义了"维度分数"（某一 DATA-BENCHMARK 组合下所有指标的平均），进度条上也显示了各维度的点位。但板块分数和 Overall Score 的计算公式中完全没有提到维度的概念，也没有说明它是"默认维度的分数"还是"所有维度的汇总平均"。

  这是计算引擎的核心问题——如果不明确，后端根本无法实现。
- **建议**：明确说明：
  1. 板块分数是按维度独立计算的还是跨维度汇总的？
  2. Overall Score 卡展示的单一数值是哪个维度的分数？还是所有维度的平均？
  3. 如果是所有维度的平均，那"四个板块的平均"和"所有维度的平均"如何嵌套？

#### 6. Snapshot 模式雷达图"各月的算术平均"与单月快照定义矛盾

- **位置**：§4.3 第 280-281 行
- **原文**：「Snapshot 模式：数据范围：单月。计算方式：每个角的值 = 该指标在各月的算术平均」
- **问题**：Snapshot 是单月快照视图，用户选一个月份看当月数据。但雷达图的计算方式写的是"各月的算术平均"——这意味着要对多个月取平均，与"单月"定义矛盾。如果是单月，每个角就应该是该月该指标的百分位值，无需跨月平均。
- **建议**：确认 Snapshot 雷达图是否只用选定月份的百分位（即每个角 = 该指标该月在当前维度下的百分位）。若确实需要多月平均，说明平均的月份范围和原因。

### 🟡 重要问题（建议修复后再进入设计）

> 描述模糊、边界未覆盖、可能有歧义——不修复则开发需要反复确认

#### 1. 基准平台名称不一致：Benchmark.it vs Benchmark.ai

- **位置**：§4.1.2 第 105 行 vs §4.2.1 第 232 行
- **原文**：
  - 第 105 行：「Internal Peers, KeyBanc, High Alpha, Benchmark.it」
  - 第 232 行：「Internal Peers → KeyBanc → High Alpha → Benchmark.ai」
- **问题**：同一个平台在文档中出现两种名称（.it 和 .ai），第 415 行也用的 Benchmark.it。开发在代码中建枚举时不知道用哪个。
- **建议**：全文统一为正确的平台名称。

#### 2. Monthly Runway 的 N/A 分类存在未覆盖的情况

- **位置**：§5.1.4 第 343-358 行
- **原文**：N/A 规则只定义了两类：
  - Top Rank（P100）：Cash ≥ 0 且 Net Burn ≥ 0
  - Bottom Rank（P0）：Cash < 0 且 Net Burn < 0
- **问题**：四象限中还有两种情况未覆盖：
  - Cash < 0 且 Net Burn ≥ 0（无现金但不亏损/盈利）→ 公式结果为负数或零，业务含义不明
  - Cash ≥ 0 且 Net Burn < 0（有现金且亏损）→ 公式 `-(Cash / NegativeBurn)` 为正值，虽然可以正常计算，但建议文档显式说明此情况是正常计算、不触发 N/A
  
  此外，第 343 行额外规定"一个为负数另一个为 0"时 P0，与表格的"≥0 且 ≥0 → P100"存在边界重叠（Cash=0 且 Burn=0 同时满足两条规则）。
- **建议**：用完整的四象限表格覆盖所有 Cash (+/-/0) × Net Burn (+/-/0) 的组合，每种组合给出明确处理方式。

#### 3. ARR Growth Rate 分母为零未处理

- **位置**：§5.1.1 第 317 行
- **原文**：「ARR Growth Rate_t = (ARR_t - ARR_(t-1)) / ARR_(t-1)」
- **问题**：若 ARR_(t-1) = 0（例如新公司首期），公式除以零。文档未说明此时显示什么、百分位如何处理。
- **建议**：补充"若 ARR_(t-1) = 0，该指标显示 N/A，排除出板块分数和 Overall Score 计算"。

#### 4. Sales Efficiency Ratio 分母为零或为负未处理

- **位置**：§5.1.6 第 371 行
- **原文**：「Sales Efficiency Ratio = (S&M Expenses + S&M Payroll) / New MRR LTM」
- **问题**：若 New MRR LTM = 0（新客户获取为零）或为负数（MRR 收缩），公式除以零或结果为负。文档未说明这些情况的处理方式。
- **建议**：补充 New MRR LTM ≤ 0 时的 N/A 规则和百分位排除逻辑。

#### 5. 外部基准 Segment 匹配无回退规则

- **位置**：§6.1.2 第 418 行
- **原文**：「匹配时应对应正确年份的相同 Segment Value 范围的百分位」
- **问题**：若该公司的 ARR 不在任何 Segment Value 范围内，或者对应年份没有该平台的数据，文档未说明回退策略。是显示 N/A？还是用最接近的 Segment？
- **建议**：补充匹配不到 Segment 时的回退规则（建议显示 N/A 并在 Tooltip 中说明原因）。

#### 6. "closed month" 概念未定义

- **位置**：§4.2.2 第 240 行
- **原文**：「默认6个月，从 closed month 往前数6个月」
- **问题**：文档其他地方用的是"最后一个有 Actual 数据的月份"或"当前月"，"closed month" 这一术语首次出现且未定义。开发不确定它与"最后有 Actual 数据的月份"是否等价。
- **建议**：用文档中已有的术语替换，或在术语表中明确定义 closed month。

#### 7. 默认展示单条分布条时未说明取哪个 DATA-BENCHMARK 组合

- **位置**：§4.2.1 第 224 行
- **原文**：「板块内每个指标默认展示一个 DATA-BENCHMARK 分布条，如果一个指标存在多个分布条，那就显示 Show All 按钮」
- **问题**：用户选了多个 DATA 和 BENCHMARK（比如 2x2=4 个组合），默认只显示一条——显示哪个组合的？是排序第一个（Actuals-Internal Peers）？还是其他规则？
- **建议**：明确默认显示的是排序最靠前的组合（即按第 231-233 行定义的顺序取第一个），或者根据其他规则确定。

#### 8. 同行匹配时目标公司当月无 ARR 数据的处理未说明

- **位置**：§6.2.1 第 452 行
- **原文**：「ARR 规模（查看哪个月用哪个月的 ARR）」
- **问题**：如果用户选择的月份目标公司没有 ARR 数据（例如选择了未来月份的 Forecast 但没有 ARR 预测），同行匹配用什么 ARR 范围？是回退到最近有数据的月份？还是无法匹配？
- **建议**：补充 ARR 缺失时的同行匹配回退规则。

#### 9. 日历默认值逻辑在注释与正文间不一致

- **位置**：§3.1 第 42 行注释 vs §4.1.3 第 116 行
- **原文**：
  - 第 42 行注释：「手动情况是最后一个有数据月份，自动情况下，15号之前用上上个月，15号之后用上个月」
  - 第 116 行：「默认值：最后一个有 Actual 数据的月份（新建公司则为当前日历月）」
- **问题**：注释中提到"手动 vs 自动"以及"15号"分界线的逻辑，但正文只写了一种简单规则。注释内容是否应纳入正文？如果"自动情况"指的是某种自动导入机制，需要在正文中正式说明。
- **建议**：要么将注释中的完整逻辑纳入正文（区分手动录入和自动导入两种场景），要么删除注释，统一使用正文的简单规则。

#### 10. Gross Margin 取值范围限制可能不准确

- **位置**：§5.1.2 第 327 行
- **原文**：「取值范围：0% 至 100%」
- **问题**：如果 Gross Profit 为负（COGS > Revenue），Gross Margin 可以为负值。限制为 0%-100% 意味着负毛利的公司无法正确表示。
- **建议**：确认是否真的限制为 0%-100%。如果是，说明负毛利的处理方式（按 0% 计？显示 N/A？）。如果不限制，移除此行或改为"通常为 0%-100%，可能为负"。

#### 11. 导出功能仅一句话，无详细描述

- **位置**：§3.1 第 66 行
- **原文**：「导出报告或截图（可选）」
- **问题**：导出是什么格式？PDF/Excel/PNG？导出内容包含哪些区域？哪些筛选条件？权限要求？如果是"可选"功能，是否在 v2 范围内？
- **建议**：若在当前版本范围内，补充导出功能的详细描述；若不在范围内，明确标注"v2 不含，计划在 vX 实现"。

### 🔵 改进建议（可在开发过程中完善）
