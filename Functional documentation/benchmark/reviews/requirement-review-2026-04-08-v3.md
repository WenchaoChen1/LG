### 功能理解

Benchmark 是 LG 系统财务模块中的对标分析功能，帮助用户将自身公司的 6 个核心财务指标（ARR Growth Rate、Gross Margin、Monthly Net Burn Rate、Monthly Runway、Rule of 40、Sales Efficiency Ratio）与内部同行（Internal Peers）和外部基准平台（KeyBanc、High Alpha、Benchmark.it）进行百分位排名比较。

核心逻辑链：用户选择筛选条件（VIEW/FILTER/DATA/BENCHMARK）→ 系统计算每个指标在每个 DATA x BENCHMARK 维度下的百分位 → 聚合为板块分数和 Overall Score → 以 Snapshot（快照）或 Trend（趋势折线）两种视图展示，底部辅以雷达图汇总。

同行匹配基于 4 维属性（公司类型、阶段、会计方法、ARR 规模），不足 4 个同行时回退到 LG 平台基准。外部基准通过线性插值或边界值从 P25/P50/P75 数据点推算百分位。

### 🔴 阻断性问题（必须修复才能进入设计）

> 逻辑矛盾、规则缺失、公式错误、关键信息丢失 —— 不修复则开发无法动工

#### 1. 默认月份逻辑存在两套规则，互相矛盾

- 位置：§3.1（行 41）
- 原文：主文「最后一个有Actual数据的月份（新建公司则为当前月）」；行内注释「手动情况是最后一个有数据月份，自动情况下，15号之前用上上个月，15号之后用上个月」
- 问题：两条规则描述了不同的默认月份选择逻辑。开发不知道该实现哪一条。"自动情况"和"手动情况"的区分条件也未定义。
- 建议：明确一条规则。如果确实存在手动/自动两种场景，需定义触发条件：什么情况下走"手动"（最后有数据月份），什么情况下走"自动"（15 号分界线）。删除行内注释，将确定的规则写入正式描述。

#### 2. FILTER 的 ALL 选项交互行为矛盾

- 位置：§3.1（行 48）vs §4.1.2（行 102）
- 原文 A（行 48）：「ALL与其他选项互斥，选中ALL后如果选择其他选项，ALL就自动取消勾选」
- 原文 B（行 102）：「ALL选中时其他自动选中，取消ALL时其他自动清空」
- 问题：这是两种完全不同的交互模式。模式 A 中 ALL 与具体选项互斥（选 GROWTH 则 ALL 取消）；模式 B 中 ALL 是"全选快捷键"（选 ALL 则 GROWTH/EFFICIENCY/MARGINS/CAPITAL 全部亮起）。开发无法判断应实现哪一种。
- 建议：选定一种模式并在两处统一。推荐模式 B（ALL 作为全选/全清快捷键），因为它允许用户从"全部"出发取消个别分类，交互更自然。

#### 3. 指标排序方向规则与个体指标定义矛盾

- 位置：§6.1.1（行 390-392）vs §5.1.1-§5.1.6 各指标定义
- 原文：§6.1.1「大部分指标按升序排序（数值小的排名靠前）」
- 问题：升序排序意味着数值小 → 排名靠前 → 百分位高。但 ARR Growth Rate（§5.1.1）、Gross Margin（§5.1.2）、Monthly Runway（§5.1.4）、Rule of 40（§5.1.5）均定义为"正向指标（数值越高，百分位排名越高）"，应使用降序排序。实际仅 Monthly Net Burn Rate 和 Sales Efficiency Ratio 适合升序排序。「大部分指标按升序排序」在 6 个指标中仅 2 个成立，与实际矛盾。
- 建议：删除「大部分指标按升序排序」的概括性描述。改为逐指标列出排序方向的明确对照表：

| 指标 | 方向 | 排序 | 含义 |
|------|------|------|------|
| ARR Growth Rate | 正向 | 降序 | 增长快 → 百分位高 |
| Gross Margin | 正向 | 降序 | 利润率高 → 百分位高 |
| Monthly Net Burn Rate | 反向 | 升序 | 烧钱少 → 百分位高 |
| Monthly Runway | 正向 | 降序 | 跑道长 → 百分位高 |
| Rule of 40 | 正向 | 降序 | 得分高 → 百分位高 |
| Sales Efficiency Ratio | 反向 | 升序 | 获客成本低 → 百分位高 |

#### 4. Monthly Net Burn Rate 的"方向"标签与实际排序行为矛盾

- 位置：§5.1.3（行 334-335）
- 原文：「方向：正向指标」紧接着「排序方式：正向排序（按数值大小从小到大排序，数值小的排名靠前，百分位更高）」
- 问题：其他"正向指标"（如 ARR Growth Rate）定义为"数值越高，百分位越高"。但 Monthly Net Burn Rate 的实际行为是"数值越低，百分位越高"（烧钱少更好），应归为反向指标。"正向指标"标签与实际排序行为矛盾，且与 §5.1.6 Sales Efficiency Ratio（同样是数值越低越好，正确标记为"反向指标"）不一致。
- 建议：将 Monthly Net Burn Rate 的方向改为「反向指标（数值越低，即现金消耗越少，百分位排名越高）」，与 Sales Efficiency Ratio 保持一致的标注规范。

#### 5. 外部基准超范围计算规则：§6.1.2 与 §7.5 矛盾

- 位置：§6.1.2（行 434）vs §7.5（行 575-576）
- 原文 A（§6.1.2）：「大于P75，展示＞P75，计算时用P100」
- 原文 B（§7.5）：「高于最高值：显示 >P75，汇总计算使用 P75」
- 问题：同一场景（指标值超出 P75 上限），§6.1.2 规定汇总时用 P100，§7.5 规定用 P75。二者差异 25 个百分点，直接影响 Overall Score 计算结果。开发不知道该信哪个。
- 建议：统一为一套规则。推荐使用 §6.1.2 的阶梯规则（更精细），并删除 §7.5 中的简化描述；或若选择 §7.5 的保守方案（用边界值），则更新 §6.1.2 使其一致。同时将 §6.1.2 的平铺规则改写为互斥区间形式（见下方第 10 条）。

#### 6. Overall Score 和板块分数未定义"多维度时用哪个维度的百分位"

- 位置：§4.1.4（行 124）、§6.4.1-§6.4.2
- 原文：「板块分数 = 该板块所有有效指标的百分位数的算术平均」「Overall Score = 四个板块分数的算术平均」
- 问题：当用户同时选中多个 DATA（如 Actuals + Committed Forecast）和多个 BENCHMARK（如 Internal Peers + KeyBanc）时，每个指标有最多 12 个不同的百分位值（每个 DATA x BENCHMARK 维度一个）。板块分数用哪个维度的百分位来计算？Overall Score 显示的单一数值对应哪个维度？文档完全未说明。
- 建议：明确定义：(a) Overall Score 的数值代表哪个维度（如默认维度 Actuals-Internal Peers），还是所有维度的均值；(b) 板块分数是按维度分别计算（每个维度一个板块分数）还是跨维度聚合。进度条上的多个点位已定义为维度分数，但 Overall Score 数值本身的计算规则缺失。

#### 7. Monthly Runway 公式的象限覆盖不完整

- 位置：§5.1.4（行 338-358）
- 原文：N/A 规则仅覆盖 (Cash >= 0, Burn >= 0) → Top 和 (Cash < 0, Burn < 0) → Bottom
- 问题：以下场景未覆盖：
  - **Cash >= 0, Burn = 0**：公式分母为零，除零错误。这是盈亏平衡公司的常见场景。
  - **Cash < 0, Burn > 0**（负现金但盈利中）：不满足任何 N/A 条件，公式结果 = -(负/正) = 正数。语义上是"还需 X 个月恢复正现金"，但这与 Runway（能撑多久）的含义不同。
  - **Cash = 0, Burn > 0**：Runway = 0 个月，进入正常排名。0 个月 Runway 是否应视为特殊情况？
- 建议：补充完整的四象限 + 零值边界矩阵：

| Cash | Burn | 处理 |
|------|------|------|
| >= 0 | > 0 | N/A，Top Rank（盈利且有现金） |
| >= 0 | = 0 | N/A，Top Rank（盈亏平衡，无破产风险） |
| >= 0 | < 0 | 正常计算 Runway |
| < 0  | > 0 | 定义为"恢复期"，建议 N/A 或单独标注 |
| < 0  | = 0 | P0（行 343 已覆盖） |
| < 0  | < 0 | N/A，Bottom Rank（已覆盖） |
| = 0  | < 0 | P0（行 343 已覆盖） |

### 🟡 重要问题（建议修复后再进入设计）

> 描述模糊、边界未覆盖、可能有歧义 —— 不修复则开发需要反复确认

#### 8. ARR Growth Rate 除零未处理

- 位置：§5.1.1（行 317）
- 原文：「ARR Growth Rate_t = (ARR_t - ARR_(t-1)) / ARR_(t-1)」
- 问题：当 ARR_(t-1) = 0（如新建公司第一期），分母为零。文档未定义此场景的处理方式。
- 建议：补充规则，如「ARR_(t-1) = 0 时，ARR Growth Rate 显示 N/A，排除出百分位计算」。

#### 9. Sales Efficiency Ratio 除零未处理

- 位置：§5.1.6（行 371）
- 原文：「Sales Efficiency Ratio = (S&M Expenses + S&M Payroll) / New MRR LTM」
- 问题：当 New MRR LTM = 0（如尚无新客收入），分母为零。
- 建议：补充规则，如「New MRR LTM = 0 时，Sales Efficiency Ratio 显示 N/A」。

#### 10. §6.1.2 超范围/插值规则的优先级不明确

- 位置：§6.1.2（行 434）
- 原文：平铺列出「大于P25→用P50」「大于P50→用P75」「大于P75→用P100」...
- 问题：一个大于 P75 数据点的值同时满足"大于P25""大于P50""大于P75"三条规则。文档未说明优先级。虽然意图显然是取最高匹配，但平铺写法容易误读。
- 建议：改写为互斥区间形式：

| 指标值所在范围 | 展示 | 汇总计算用值 |
|--------------|------|------------|
| < P25 数据点 | <P25 | P0 |
| = P25 数据点 | P25 | P25 |
| P25 ~ P50 之间 | ~Pxx（插值） | 插值结果 |
| = P50 数据点 | P50 | P50 |
| P50 ~ P75 之间 | ~Pxx（插值） | 插值结果 |
| = P75 数据点 | P75 | P75 |
| > P75 数据点 | >P75 | P100（或 P75，需与 §7.5 统一） |

#### 11. 雷达图 Snapshot 模式描述矛盾

- 位置：§4.3（行 280-281）
- 原文：「Snapshot 模式：数据范围：单月 / 计算方式：每个角的值 = 该指标在各月的算术平均」
- 问题：Snapshot 是单月视图（用户通过日历选择一个月），但计算方式写的是"各月的算术平均"。单月不存在"各月"可供平均。疑似从 Trend 模式复制时未修改。
- 建议：改为「Snapshot 模式：每个角的值 = 该指标在选定月份的百分位值」。

#### 12. 外部基准匹配无年份/ARR 范围缺失时的回退规则

- 位置：§6.1.2（行 418）
- 原文：「匹配时应对应正确年份的相同Segment Value范围的百分位」
- 问题：如果外部基准数据中没有目标公司所在 ARR 范围或对应年份的数据，系统如何处理？没有回退规则。
- 建议：补充回退策略，如「若外部基准无匹配的 Segment Value，该基准源对应维度显示 N/A；若无匹配年份，使用最近可用年份并标注」。

#### 13. DATA 和 BENCHMARK 的最低选择数量未定义

- 位置：§4.1.2（行 103-104）
- 原文：DATA 和 BENCHMARK 均为"多选"，但未定义是否允许全部取消选择。
- 问题：如果用户取消所有 DATA 选项或所有 BENCHMARK 选项，页面应显示什么？空白？错误提示？
- 建议：补充规则，如「DATA 和 BENCHMARK 至少需选择一项。若用户尝试取消最后一项，保持该项选中状态」。

#### 14. Trend 视图 Percentile/Amount 切换控件未在筛选区描述

- 位置：§4.2.2（行 253）
- 原文：「折线图的展示有Percentile和Amount两个选项」
- 问题：这是一个新的 UI 控件，但未在 §4.1.2 筛选条件区域中描述。该切换按钮的位置、样式、交互模式（单选？）、默认值都缺失。
- 建议：在 §4.1.2 或 §4.2.2 中补充该控件的完整描述：位置（每个板块内还是全局）、默认值（Percentile）、切换行为。

#### 15. "Closed month" 术语未定义

- 位置：§4.2.2（行 240）
- 原文：「默认6个月，从closed month 往前数6个月」
- 问题："Closed month" 在文档中首次出现且未定义。是否等同于 §4.1.3 的"最后一个有Actual数据的月份"？
- 建议：统一使用已定义的术语，或在首次使用处给出定义。

#### 16. Gross Margin 取值范围与现实不符

- 位置：§5.1.2（行 327）
- 原文：「取值范围：0% 至 100%」
- 问题：Gross Margin = Gross Profit / Gross Revenue。当 COGS > Revenue 时 Gross Profit 为负，Gross Margin 可以为负值。限制在 0%-100% 会掩盖亏损公司的真实情况。
- 建议：如果业务上确实限制为 0%-100%（将负值视为 0%），需明确说明截断规则和原因。否则应将取值范围改为「不限（可为负值）」。

#### 17. Rule of 40 公式使用 MRR YoY Growth Rate，与 §5.1.1 的 ARR Growth Rate 术语不一致

- 位置：§5.1.5（行 363）
- 原文：「Rule of 40 = (Net Profit Margin + MRR YoY Growth Rate) x 100%」
- 问题：§5.1.1 定义的是 ARR Growth Rate，但 Rule of 40 使用 MRR YoY Growth Rate。MRR（月度）和 ARR（年度）虽然增长率理论上相同，但术语不一致会让开发产生疑问：是否需要单独计算 MRR YoY Growth Rate？
- 建议：如果确实使用同一个增长率，统一术语为 ARR Growth Rate 并注明引用。如果是不同计算，需在 §5.1 中补充 MRR YoY Growth Rate 的公式定义。

#### 18. LG 平台基准的定义缺失

- 位置：§6.2.2（行 465）、§6.4.5（行 532）
- 原文：§6.2.2「回退到 LG 平台基准」；§6.4.5「降级到所有活跃LG企业的Actual」
- 问题：「LG 平台基准」是否等同于「所有活跃 LG 企业」？如果是，是否忽略公司类型、阶段、ARR 范围等匹配条件？仅保留排除条件（Exited/Shut Down/Inactive）？文档未明确。
- 建议：在 §6.2.2 中明确定义：「LG 平台基准 = 所有状态为 Active 的 LG 平台公司（不限公司类型、阶段、ARR 范围），仅排除 Exited/Shut Down/Inactive」（或其他具体规则）。

#### 19. 指标 Tooltip 悬停内容描述不完整

- 位置：§4.2.1（行 195-196）
- 原文：「各指标名称悬停内容：指标内容，右侧加指标标签（Guess/Exact），指标公式，Segment,Source。指标公式在下面各指标信息中包含」
- 问题：「Guess/Exact」标签的含义未定义 —— 什么条件下显示 Guess？什么条件下显示 Exact？「Segment」和「Source」的具体内容和来源也未说明。
- 建议：补充 Guess/Exact 标签的判定规则，以及 Segment（ARR 范围？）和 Source（数据来源？）字段的取值范围。

### 🔵 改进建议（可在开发过程中完善）

#### 20. §1.2 产品目标与 §1.1 功能定义完全重复

- 位置：§1.2（行 15）
- 问题：产品目标应描述可衡量的业务目标（如"帮助投资组合公司识别改进方向，提升 XX 指标"），而非重复功能描述。
- 建议：重新定义产品目标为可衡量的 KPI，或删除此小节。

#### 21. 进度条宽度描述不可测试

- 位置：§4.1.4（行 129）
- 原文：「占卡片宽度的60-70%」
- 问题：60-70% 是一个范围，QA 无法验证。
- 建议：确定一个具体百分比（如 65%），或改为「约占卡片宽度的 2/3」并允许设计师微调。

#### 22. 日历选择区域位置不确定

- 位置：§4.1.3（行 114）
- 原文：「筛选卡片下方或右侧」
- 问题：两个位置意味着布局不确定，前端无法确定实现。
- 建议：参照截图确定一个位置。从截图看，日历在筛选卡片的右侧。

#### 23. 无性能要求

- 问题：§8.1 所有操作反馈方式标注"即时"，但未定义可接受的响应时间。当同行数量多或数据量大时，"即时"可能无法实现。
- 建议：补充性能要求，如「指标计算和页面刷新应在 2 秒内完成」。

#### 24. 缺少错误状态描述

- 问题：文档未描述网络失败、API 超时、数据加载异常时的用户体验。
- 建议：补充错误状态说明，如 Loading 状态、错误提示内容、重试机制。

#### 25. 缺少权限模型

- 问题：文档未说明谁能看到 Benchmark 数据。所有用户？仅管理员？公司级隔离？
- 建议：补充权限说明，至少明确数据可见性范围。

#### 26. 导出功能缺少细节

- 位置：§3.1（行 65）
- 原文：「导出报告或截图（可选）」
- 问题：导出格式（PDF/CSV/PNG？）、导出内容范围、触发按钮位置均未定义。
- 建议：如为 V2 范围外功能，标注"待后续版本定义"；如在范围内，补充完整描述。

#### 27. 行内注释应正式化

- 位置：§3.1（行 41）、§4.2.1（行 195-196、222-224）等多处
- 问题：文档中混有行内备注/评审意见（如行 41 的"手动情况..."），格式上与正式描述混淆，不易区分是最终规则还是讨论笔记。
- 建议：将所有行内注释整合为正式描述，或移入"待确认问题"专节。

#### 28. §6.2.2 同行回退表格存在重复行

- 位置：§6.2.2（行 465-466）
- 原文：「无符合条件的同行 | 找不到 4 个及以上同行」和「同行不足4个 | 同行公司 < 4 个」
- 问题：两行描述的是同一条件（同行 < 4），属于重复。
- 建议：合并为一行。

### 结论

**是否可进入设计阶段**：修复 7 个阻断性问题后可以

**必须优先修复的 Top 3 问题**：

1. **指标排序方向规则矛盾**（问题 3 + 4）—— 这是百分位排名引擎的核心逻辑，排错方向则所有指标百分位都会反转。必须提供明确的逐指标排序方向对照表。
2. **Overall Score / 板块分数的多维度计算规则缺失**（问题 6）—— 当用户选多个 DATA x BENCHMARK 时，系统显示的核心数值（Overall Score）用哪个维度的百分位？不定义则 UI 和后端无法对齐。
