### 功能理解

Benchmark 是 LG 系统财务模块中的对标分析功能，帮助用户将自身公司的 6 个核心财务指标（ARR Growth Rate、Gross Margin、Monthly Net Burn Rate、Monthly Runway、Rule of 40、Sales Efficiency Ratio）与内部同行（Internal Peers）和外部基准平台（KeyBanc、High Alpha、Benchmark.it）进行百分位排名比较。

核心逻辑链：用户选择筛选条件（VIEW / FILTER / DATA / BENCHMARK + 日历月份）→ 系统为每个指标在每个 DATA x BENCHMARK 维度下计算百分位 → 聚合为板块分数（4 板块）和 Overall Score → 以 Snapshot（快照百分位条）或 Trend（趋势折线）两种视图展示，底部辅以六角雷达图汇总。

同行匹配基于 4 维属性（公司类型、阶段、会计方法、ARR 规模区间），不足 4 个时回退到 LG 平台全量活跃公司。外部基准通过线性插值或边界值从 P25/P50/P75 数据点推算百分位。

#### §三 完整业务流程（2 个）

1. **默认月份逻辑存在两套互斥规则**
   - 位置：§3.1（行 42）
   - 原文 A：「最后一个有Actual数据的月份（新建公司则为当前月）」
   - 原文 B（行内注释）：「手动情况是最后一个有数据月份，自动情况下，15号之前用上上个月，15号之后用上个月」

2. **FILTER 的 ALL 选项交互行为矛盾**
   - 位置：§3.1（行 48-49）vs §4.1.2（行 103）
   - 原文 A（行 48-49）：「ALL与其他选项互斥，选中ALL后如果选择其他选项，ALL就自动取消勾选，其他选项可同时选中」
   - 原文 B（行 103）：「ALL选中时其他自动选中，取消ALL时其他自动清空」
   - 问题：两种完全不同的交互模式。模式 A：ALL 与具体选项互斥（选 GROWTH 则 ALL 取消）。模式 B：ALL 是全选快捷键（选 ALL 则四项全亮，取消 ALL 则全灭）。开发无法判断实现哪种。
   - 建议：选定一种模式并在两处统一描述。推荐模式 B 并补充完整交互矩阵：(a) 点击 ALL → 四项全选中；(b) 取消 ALL → 四项全清空；(c) 从全选状态取消某一项 → ALL 自动取消，其余不变；(d) 从部分选中状态全部勾选 → ALL 自动选中。

#### §四 功能详细描述（1 个）

3. **Overall Score 和板块分数未定义多维度时使用哪个维度的百分位**
   - 位置：§4.1.4（行 124-126）、§6.4.1-§6.4.2
   - 原文：「板块分数 = 该板块所有有效指标的百分位数的算术平均」「Overall Score = 四个板块分数的算术平均」

#### §五 核心指标和公式（2 个）

4. **Monthly Net Burn Rate 的方向描述与排序行为自相矛盾**
   - 位置：§5.1.3（行 334-336）
   - 原文：「方向：正向指标」「排序方式：正向排序（按数值大小从小到大排序，数值小的排名靠前，百分位更高）」
   - 问题：用百分位公式 P = ((R-1)/(N-1)) x 100% 验证：升序排序时，rank 1 = 最小值 → P = 0%（最低百分位），rank N = 最大值 → P = 100%（最高百分位）。因此升序排序意味着**数值越大，百分位越高**，而非「数值小的……百分位更高」。描述文本与数学公式直接矛盾。此外，"正向指标"的含义应为"数值越高越好"（与 ARR Growth Rate 等一致），但描述却暗示数值越低越好。
   - 建议：修正描述为「排序方式：升序排序（数值越大 → 排名号越大 → 百分位越高）」。如果业务意图确实是"数值越低越好"（即传统 burn rate 含义），则应改标签为"反向指标"并使用降序排序，同时重新审视公式 Burn Rate = Net Income - Cap R&D 的语义。

5. **Monthly Runway 公式的象限覆盖不完整**
   - 位置：§5.1.4（行 338-358）
   - 原文：N/A 规则仅覆盖 (Cash >= 0, Burn >= 0) → Top Rank 和 (Cash < 0, Burn < 0) → Bottom Rank，以及「当cash或Monthly Burn Rate其中一个为负数并且另一个为0的时候，显示NA值，排名最差，得P0」
   - 问题：以下场景未覆盖：
     - **Cash >= 0, Burn = 0**：公式 Runway = -(Cash / 0) → 除零错误。这是盈亏平衡公司的常见场景。
     - **Cash < 0, Burn > 0**（负现金但盈利中）：公式结果 = -(负/正) = 正数，语义为"需 X 个月恢复正现金"，但这与 Runway（还能撑多久）的含义不同。
     - **Cash = 0, Burn < 0**（零现金且亏损）：Runway = -(0/负) = 0 个月。0 个月是否应视为特殊情况？
   - 建议：补充完整的四象限 + 零值边界矩阵：

     | Cash | Burn | 处理 | 排名 |
     |------|------|------|------|
     | >= 0 | > 0 | N/A | Top Rank（盈利且有现金） |
     | >= 0 | = 0 | N/A | Top Rank（盈亏平衡，无破产风险） |
     | >= 0 | < 0 | 正常计算 | 按公式排名 |
     | < 0  | > 0 | 定义缺失 | 建议 N/A + 单独标注"恢复期" |
     | < 0  | = 0 | N/A | P0（行 343 已覆盖） |
     | < 0  | < 0 | N/A | Bottom Rank（已覆盖） |
     | = 0  | < 0 | 正常计算 | Runway = 0，进入排名 |
     | = 0  | = 0 | N/A | Top Rank（无现金也无消耗） |

#### §六 百分位排名引擎（3 个）

6. **排序方向的概括性描述与正向指标实际行为矛盾**
   - 位置：§6.1.1（行 390-392）vs §5.1.1-§5.1.6 各指标定义
   - 原文：「大部分指标按升序排序（数值小的排名靠前）」
   - 问题：结合公式 P = ((R-1)/(N-1)) x 100%，升序排序使得**数值大的百分位高**。这对 4 个正向指标（ARR Growth Rate、Gross Margin、Monthly Runway、Rule of 40）是正确的。但括号内「数值小的排名靠前」可能被误读为"数值小 = 百分位高"（实际相反）。更严重的问题：文档未说明反向指标（Sales Efficiency Ratio——数值越低越好）应使用降序排序还是修改公式。6 个指标中仅 1 个（或 2 个，取决于 Burn Rate 的方向定义）是反向指标，「大部分按升序」对正向指标成立，但反向指标的排序规则完全缺失。
   - 建议：(a) 删除模糊的概括性描述；(b) 增加逐指标排序方向对照表：

     | 指标 | 方向 | 排序 | 含义 |
     |------|------|------|------|
     | ARR Growth Rate | 正向 | 升序 | 增长快 → 百分位高 |
     | Gross Margin | 正向 | 升序 | 利润率高 → 百分位高 |
     | Monthly Net Burn Rate | 待确认 | 待确认 | 见问题 #4 |
     | Monthly Runway | 正向 | 升序 | 跑道长 → 百分位高 |
     | Rule of 40 | 正向 | 升序 | 得分高 → 百分位高 |
     | Sales Efficiency Ratio | 反向 | 降序 | 获客成本低 → 百分位高 |

7. **外部基准超范围计算规则：§6.1.2 与 §7.5 矛盾**
   - 位置：§6.1.2（行 434）vs §7.5（行 575-576）
   - 原文 A（§6.1.2）：「大于P75，展示 >P75，计算时用P100」「小于P25，展示 <P25，计算时用P0」
   - 原文 B（§7.5）：「高于最高值：显示 >P75，汇总计算使用 P75」「低于最低值：显示 <P25，汇总计算使用 P25」
   - 问题：同一场景（指标值超出基准范围），§6.1.2 规定上限用 P100、下限用 P0，§7.5 规定用边界值 P75/P25。上限差异 25 个百分点，直接影响 Overall Score。开发不知道该信哪个。
   - 建议：统一为一套规则。§6.1.2 的阶梯规则更精细（有 >P25/P50/P75 分级），建议以此为准，删除 §7.5 中的简化描述。同时将 §6.1.2 的平铺规则改写为互斥区间表（见问题 #14）。

8. **"所有指标值完全相同"处理方式矛盾**
   - 位置：§6.1.1 步骤 4（行 412）vs §7.4（行 564-569）
   - 原文 A（§6.1.1）：「若所有指标值完全相同：不显示百分位，不计入 metric summary 和 Overall Score」
   - 原文 B（§7.4）：「分配相同的百分位值（如都是 50%ile）」「正常参与权重计算」
   - 问题：§6.1.1 说排除不计，§7.4 说给 50%ile 并正常计算。二者完全对立。按竞争排名公式，所有值相同时全部 rank = 1，P = ((1-1)/(N-1)) x 100% = 0%，也非 50%ile。三方说法均不一致。
   - 建议：选定一个规则：(a) 若排除：统一为 §6.1.1 的方案，删除 §7.4 或标注为已废弃；(b) 若包含：统一为某个具体百分位值（用公式计算为 P0，或业务决定给 P50），更新 §6.1.1。

#### §五 核心指标和公式（4 个）

15. **ARR Growth Rate 除零未处理**
    - 位置：§5.1.1（行 317）
    - 原文：「ARR Growth Rate_t = (ARR_t - ARR_(t-1)) / ARR_(t-1)」
    - 问题：当 ARR_(t-1) = 0（如新建公司第一期），分母为零，公式无法计算。
    - 建议：补充规则「ARR_(t-1) = 0 时，ARR Growth Rate 显示 N/A，排除出百分位计算」。

16. **Gross Margin 取值范围 0%-100% 与现实不符**
    - 位置：§5.1.2（行 327）
    - 原文：「取值范围：0% 至 100%」
    - 问题：Gross Margin = Gross Profit / Gross Revenue，当 COGS > Revenue 时 Gross Profit 为负，Gross Margin 可以为负值。限定在 0%-100% 会掩盖亏损公司。
    - 建议：如果业务确实要截断为 0%（将负值视为 0%），需明确说明截断规则和原因；否则改取值范围为「不限（可为负值）」。

17. **Sales Efficiency Ratio 除零未处理**
    - 位置：§5.1.6（行 371）
    - 原文：「Sales Efficiency Ratio = (S&M Expenses + S&M Payroll) / New MRR LTM」
    - 问题：当 New MRR LTM = 0（尚无新客收入），分母为零。
    - 建议：补充规则「New MRR LTM = 0 时，Sales Efficiency Ratio 显示 N/A，排除出百分位计算」。

18. **Rule of 40 使用 MRR YoY Growth Rate，与 §5.1.1 的 ARR Growth Rate 术语不一致**
    - 位置：§5.1.5（行 363）
    - 原文：「Rule of 40 = (Net Profit Margin + MRR YoY Growth Rate) x 100%」
    - 问题：§5.1.1 定义的增长率为 ARR Growth Rate，但 Rule of 40 使用 MRR YoY Growth Rate。MRR（月度）和 ARR（年度）虽然增长率理论上相同，但术语不一致会让开发产生疑问：是否需要单独计算？
    - 建议：如果使用同一个增长率，统一术语并注明引用（如「MRR YoY Growth Rate，等价于 §5.1.1 的 ARR Growth Rate」）；如果是不同计算，需补充 MRR YoY Growth Rate 的公式定义。

#### §六 百分位排名引擎（6 个）

19. **反向指标的排序方式未明确规定**
    - 位置：§6.1.1（行 390-393）
    - 原文：仅描述「大部分指标按升序排序」和一个百分位公式
    - 问题：Sales Efficiency Ratio 是反向指标（数值越低越好）。使用升序排序 + P=((R-1)/(N-1))*100% 会使低值→低百分位，与业务意图相反。文档未说明反向指标应使用降序排序或修改公式。
    - 建议：明确规定：反向指标使用降序排序（数值从大到小），使得数值低→排名号大→百分位高。或者保持升序排序但使用反向公式 P=((N-R)/(N-1))*100%。无论哪种方案，需显式声明。

20. **§6.1.2 超范围/插值规则的优先级不明确**
    - 位置：§6.1.2（行 434）
    - 原文：平铺列出「大于P25→展示>P25，计算时用P50」「大于P50→展示>P50，计算时用P75」「大于P75→展示>P75，计算时用P100」等
    - 问题：一个大于 P75 数据点的值同时满足"大于P25""大于P50""大于P75"三条规则。虽然意图是取最高匹配，但平铺写法容易误读。
    - 建议：改写为互斥区间形式：

      | 指标值所在范围 | 展示 | 汇总计算用值 |
      |--------------|------|------------|
      | < P25 数据点 | <P25 | P0 |
      | = P25 数据点 | P25 | P25 |
      | P25 ~ P50 之间 | ~Pxx（线性插值） | 插值结果 |
      | = P50 数据点 | P50 | P50 |
      | P50 ~ P75 之间 | ~Pxx（线性插值） | 插值结果 |
      | = P75 数据点 | P75 | P75 |
      | > P75 数据点 | >P75 | P100（需与 §7.5 统一） |

21. **线性插值公式除零场景未处理**
    - 位置：§6.1.2（行 429）
    - 原文：「~P = P_low + (P_high - P_low) * (d - d_low) / (d_high - d_low)」
    - 问题：当 d_high = d_low（两个基准数据点值相同，如 P25 = P50）时，分母为零。
    - 建议：补充规则「若 d_high = d_low 且 d = d_high，直接取 P_high；若 d != d_high，按超范围规则处理」。

22. **外部基准匹配无对应年份或 ARR 范围时的回退规则缺失**
    - 位置：§6.1.2（行 418）
    - 原文：「匹配时应对应正确年份的相同Segment Value范围的百分位」
    - 问题：如果外部基准数据中没有目标公司所在 ARR 范围或对应年份的数据，系统如何处理？
    - 建议：补充回退策略，如「若外部基准无匹配的 Segment Value，该基准源对应维度显示 N/A；若无匹配年份，使用最近可用年份并标注来源年份」。

23. **LG 平台基准的精确定义缺失**
    - 位置：§6.2.2（行 465）、§6.4.5（行 532）
    - 原文：「回退到 LG 平台基准」「降级到所有活跃LG企业的Actual」
    - 问题：「LG 平台基准」是否等同于「所有活跃 LG 企业」？是否忽略公司类型、阶段、ARR 范围等匹配条件？仅保留排除条件（Exited/Shut Down/Inactive）？
    - 建议：在 §6.2 中明确定义「LG 平台基准 = 所有状态为 Active 的 LG 平台公司（不限公司类型、阶段、ARR 范围），仅排除 Exited/Shut Down/Inactive」。

24. **Forecast 内部同行 "6个月连续有效数据验证" 规则模糊**
    - 位置：§6.4.6（行 536）
    - 原文：「数据源：同行企业预测ARR + 6个月连续有效数据验证（以最后一个有预测数据的月份为基准）」
    - 问题：什么是"6个月连续有效数据"？是指同行必须有最近连续 6 个月的预测数据才能纳入计算？如果同行只有 4 个月预测数据，是否被排除？"以最后一个有预测数据的月份为基准"是指从该月往前数 6 个月？
    - 建议：用具体条件重述，如「同行企业必须在从最后一个有预测数据的月份往前连续 6 个月均有有效预测数据，否则该同行在 Forecast 维度被排除」。

#### §八 UI交互（1 个）

    - 原文 B：「Peer Count: 45 / Cohort Size」
    - 问题：§8.2 在 Peer Count 后额外列出 Cohort Size，但 Cohort Size 的含义未在任何地方定义。它与 Peer Count 是否相同？如果不同，分别代表什么？
    - 建议：统一术语。如 Cohort Size 有独立含义（如包含被排除的同行），补充定义；如果是同一概念，删除重复项。

### 结论

**是否可进入设计阶段**：修复 8 个阻断性问题后可以

**必须优先修复的 Top 3 问题**：

1. **[:red_circle: §6.1.1 + §5.1.3] 排序方向规则与百分位公式矛盾**（问题 #4 + #6 + #19）—— 这是百分位排名引擎的核心逻辑。排序方向描述与公式不一致，反向指标排序方式缺失，将导致所有指标百分位计算结果不可预测。必须提供明确的逐指标排序方向对照表。
2. **[:red_circle: §4.1.4] Overall Score 多维度计算规则缺失**（问题 #3）—— 当用户选多个 DATA x BENCHMARK 时，页面核心数值（Overall Score + 板块分数）使用哪个维度的百分位？不定义则 UI 和后端无法对齐。
3. **[:red_circle: §6.1.2 vs §7.5] 外部基准超范围计算矛盾**（问题 #7）—— §6.1.2 用 P100 而 §7.5 用 P75，差异 25 个百分点，直接影响分数计算。
