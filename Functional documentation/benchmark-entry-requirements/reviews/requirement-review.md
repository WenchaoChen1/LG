## 二、逻辑问题（共 9 个）

### 🔴 严重（影响开发或逻辑矛盾）— 3 个

**L-01. [4.2/4.3 节 vs BR-14] 唯一性校验时机矛盾**

4.2 节和 4.3 节流程明确写"用户点击行内 ✓ 确认时校验 Platform + FY Period 组合"；但 BR-14 写"**保存时**若检测到冲突"。两处描述完全矛盾，开发无法确定在哪个环节实现校验。

> 建议：统一为"用户点击行内 ✓ 确认按钮时即时校验（前端校验）"，删除 BR-14 中"保存时"的表述，改为"在行内确认时检测到 Platform + FY Period 组合与同 Metric 下已有记录重复，则显示提示并阻止行提交"。

**L-02. [4.5 节 vs 4.7 节 / BR-16、BR-17] LG Formula 独立保存与批量保存的边界未定义**

Formula 走独立 API（blur 即保存），详情行走保存按钮（批量提交）。但以下问题无法从文档中找到答案：
- Formula 修改是否属于"未保存的变更"？
- 是否会触发保存按钮高亮？
- 离开页面时是否弹"有未保存的更改"提示？

> 建议：在 BR-16 和 BR-17 中补充"LG Formula 的修改走独立保存通道（blur 即刻提交到后端），不纳入批量保存队列，不触发保存按钮高亮，也不触发离开页面的保护提示"。

**L-03. [4.2/4.3 前置条件 vs BR-15] 编辑互斥规则的跨 Metric 场景和静默取消行为未定义**

BR-15 说"当用户触发新的编辑或新增操作时，系统自动取消当前正在编辑的行"，但存在两个未定义问题：
- (a) 该规则是否跨 Metric 全局生效（如 Metric A 正在编辑，用户点击 Metric B 的 Add detail）？
- (b) 若被自动取消的行已有输入内容，是否提示用户？还是静默丢弃？

> 建议：明确互斥规则全局生效（覆盖跨 Metric），自动取消为静默操作（不弹提示），同时补充"被取消的 Metric 的 Add detail 按钮同步恢复显示"。

### 🟡 一般（模糊或可能有歧义）— 4 个

**L-04. [4.7 节 / 8.2 节] 保存按钮与货币转换开关的相对顺序未明确**

两处描述均将两者放在标题行右端，但未说明谁左谁右。

> 建议：在 8.2 节标题区补充具体排列顺序。

**L-05. [BR-17] 离开页面保护的技术边界未定义**

文档写"关闭标签页、点击导航等"，但 CIOaaS 是 SPA 应用，浏览器原生 `beforeunload` 无法拦截前端路由跳转（如点击左侧菜单），需额外实现路由守卫（Router Prompt）。

> 建议：明确需同时覆盖浏览器级事件和 SPA 内部路由跳转。

**L-06. [7.2 节] 数值字段对齐/字体说明遗漏**

25th / Median / 75th 在 7.1 节已明确"右对齐、等宽字体"，但 7.2 子级字段表中缺少此说明，开发人员对照 7.2 实现时容易遗漏。

> 建议：在 7.2 子级字段表脚注中补充。

**L-07. [4.2 节 / BR-15] "Add detail" 按钮恢复的跨 Metric 联动未覆盖**

当另一个 Metric 下的新增行被 BR-15 自动取消时，该 Metric 的 "Add detail" 按钮应同步恢复显示，文档未描述。

> 建议：在 4.2 节前置条件中补充。

### 🔵 建议（可选优化）— 2 个

**L-08.** [4.4 节] 删除流程的"风险说明"可补充：在点击保存按钮之前刷新页面可恢复被删除的数据（因数据尚未提交到后端）。

**L-09.** [BR-05] Category 纵向合并行数计算规则只描述了单个 Metric 情形，建议补充"同一 Category 下所有 Metric 所占行数之和即为该 Category 单元格的 rowSpan"。

## 六、总结

### 整体评估：需修改后再用

需求文档整体质量较高，结构完整（9 章），13 个原始疑点均已确认并正确融入文档，使用场景、业务流程、字段规格详尽。主要问题集中在 **3 个严重的逻辑矛盾** 和 **14 个新增业务疑点**。

### 必须修复后才能进入设计阶段的问题（共 3 个）

1. **L-01 唯一性校验时机矛盾**：4.2/4.3 节写"行内 ✓ 确认时即时校验"，BR-14 写"保存时校验"，必须统一
2. **L-02 LG Formula 独立保存与批量保存边界未定义**：Formula 修改是否纳入批量保存队列、是否触发保存按钮高亮、是否触发离开页面保护
3. **L-03 编辑互斥（BR-15）跨 Metric 场景未定义**：必须明确全局生效且自动取消为静默操作

### 必须确认的新增疑点（共 14 个）

Q14-Q27 详见第五节和 `requirement/questions-review.md`，其中 Q14、Q15 为阻塞级，Q16-Q20 为高影响。

### 可在开发过程中确认的问题（共 4 个）

1. 保存按钮与货币转换开关的相对排列顺序（L-04）
2. Category 和 Metric Name 列是否为横向滚动固定列 sticky column（D-03）
3. 货币转换开关状态的生命周期（Q23）
4. 离开页面保护是否覆盖 SPA 内部路由跳转（L-05）
